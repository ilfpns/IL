### Nagle 알고리즘이 뭘까
[[Network]]
Nagle 알고리즘은 작은 패킷들을 모아서 한번에 보내는 알고리즘이다

3가지 단계로 진행된다

1. 보낼 데이터가 MSS (Maximum Segment Size)만큼 쌓이면 즉시 전송
2. MSS보다 작더라도 이전 패킷에 대한 ACK가 오면 다음 데이터를 전송한다
3. ACK가 올 때까지는 데이터를 버퍼에 쌓아둔다

⇒ MMS가 꽉 참 (1)

⇒ ACK가 수신됨 (2)

⇒ 1 또는 2가 만족할 때 다음 패킷 전송

**장점**

- 작은 패킷을 줄여 오버헤드를 줄인다
- 불필요한 트래픽을 완화한다

**단점**

- 속도가 느리다
    
    ⇒ 네트워크 부하를 줄이기 위해 균형점을 찾아 적용해야 한다
    
- TCP의 다른 기능인 Delayed ACK와 함께 사용될 경우 서로 응답 신호를 기다리며 최대 응답 시간이 길어질 수 있다
    
    - Delayed ACK
        
        수신 측이 패킷을 받을 때마다 매번 ACK를 보내는 대신, 조금 기다렸다가 다음 응답과 합쳐서 보내거나 데이터에 얹어서 보내는 방식이다
        
        **왜 기다림을 가질까?**
        
        - 패킷당 ACK를 보내면, 네트워크에 ACK 패킷이 너무 많아진다
        - 데이터에 함께 보낼 때 데이터에 얹어서 (piggyback) 보낼 수 있다
    
    **왜 TCP Nagle과 함께 쓰면 안 좋을까?**
    
    - Nagle : ACK올 때까지 기다림
    - Delayed ACK : ACK 모아서 보내려고 다음 데이터 기다림
    
    ⇒ 네트워크 지연
    

**MMS**

최대로 담을 수 있는 패킷의 양을 뜻한다

- 일반적으로 이더넷 환경에서는 MTU (1500바이트)에서 IP및 TCP 헤더 크기를 제외한 1460바이트로 설정된다
    
- MTU
    
    한번에 보낼 수 있는 가장 큰 데이터 패킷의 크기
    
    **의문점**
    
    ⇒ 그렇다면 MTU보다 큰 데이터는 항상 전송을 못 할까?
    
    그렇지 않다, 이 경우 데이터 단편화를 거친다
    
    MTU는 100이고 보내려 하는 데이터가 110일 경우 :
    
    - 첫 번째 패킷: [IP 헤더(20) + TCP 헤더(20) + 데이터(60)] = 총 100바이트 (MTU 꽉 채움)
    - 두 번째 패킷: [IP 헤더(20) + TCP 헤더(20) + 남은 데이터(50)] = 총 90바이트

|**구분**|나글 알고리즘 사용 (ON)|나글 알고리즘 중단 (OFF)|
|---|---|---|
|적합한 사례|파일 전송(FTP), 대용량 데이터 전송|실시간 게임, 주식 거래, 마우스 입력|
|핵심 가치|처리량(Throughput) 극대화|빠른 응답성(Low Latency)|

### 왜 TCP에 Nagle 알고리즘이 필요할까?

⇒ 1980년대에 대화형 서비스에서 1바이트를 전송하기 위해 TCP헤더 (20바이트) IP헤더 (20바이트)를 붙이며, 총 41바이트를 전송하는 낭비를 겪음

⇒ Nagle방법을 사용하여 해결
