### 스케줄링이 뭘까

⇒ 한정된 자원을 언제, 누구에게 분배할지 결정하는 계획 과정을 의미한다

- 언제 : time
- 누구 : process ( or task)

**왜 스케줄링을 할까?**

실제로 Core가 1개인 컴퓨터는 동시에 여러 Process를 진행시키지 않는다

실제로는 아주 짧은 시간동안 여러 프로그램을 번갈아 가며 실행된다

- FIFO : 먼저 들어온 프로세스를 먼저 실행함
- SJF : 빨리 끝낼 수 있는 프로그램 먼저 처리
- RR : 공평한 time slice를 주며, 일을 처리하지 못한 프로세스는 뒤로 보냄

### 스케줄링은 중간에 바뀔 수 있을까?

한번 스케줄링의 선택을 받은 프로세스는 어떠한 경우에도 일을 끝날 때까지 자원을 놓아주지 않는다

다음과 같은 경우를 비선점 스케줄링 이라고 한다

**비선점 스케줄링 (Non-Preemptive)**

자원을 가진 프로세스가 작업이 끝날 때까지 강제로 권한을 뺏을 수 없다

- 특징: 순서대로 처리한다, 강제로 뺏는 일이 없다
- 장점: 스케줄링 자체가 단순하고, 순서가 바뀌는 혼란(오버헤드)이 적다
    - 오버헤드란?
        
        ⇒ 계산된 소모 비용보다 더 큰 비용을 발생시키는 상황
        
- 단점: 아주 짧은 작업이라도 앞에 긴 작업이 있으면 무한정 기다려야 한다

**선점 스케줄링 (Preemptive)**

비선점 스케줄링과 달리 급한 작업이 있을 시 다른 프로세스의 자원을 뺏어올 수 있다

- 특징: 우선순위가 높은 급한 일이 들어오면 현재 작업을 중단시키고 새 작업을 먼저 처리한다
- 장점: 응답 속도가 빨르.
- 단점: Context Switcing 과정에서 시스템에 부하가 생길 수 있다

### Context Switching

CPU가 여러 프로세스를 번갈하 실행하며, 실제로는 동시에 실행 되듯 보이게 한다

CPU 내부에는 PCB(Process Control Block)라는 프로세스 정보 블럭이 있다

Flow

1. 상태 저장: 실행 중이던 프로세스 A의 정보(어디까지 읽었는지, 변수 값은 무엇인지 등)를 PCB에 저장한다
2. 상태 복구: 새로 실행할 프로세스 B의 정보를 해당 PCB에서 CPU로 불러온다
3. 실행: 이제 CPU는 프로세스 B의 중단되었던 지점부터 다시 일을 시작한다

이런 Context Switching은 큰 단점이 있다

컨텍스트 스위칭이 일어나는 동안 CPU는 실제 아무런 일을 하지 못합니다.

- 오버헤드 발생: 스위칭이 너무 자주 일어나면 중간에 연산을 하지 못하므로 컴퓨터가 느려진다

### 스케줄링 정책 (Policy)와 클래스 (Class)란?

OS에서는 성격이 전혀 다른 수많은 프로세스가 돌아간다. 이러한 프로세스들을 전부 같은 규칙으로 처리하는 것은 효율적이다

**클래스**

- 프로세스들을 중요도나 성격에 따라 묶어둔 그룹이다
- 실시간 클래스, 인반 클래스 등으로 나눌 수 있다

**정책**

- 각 클래스에 맞게 CPU를 분배하는 방식
- Linux에서는 다음과 같은 정책이 있다
    - SCHED_FIFO : Real-Time 정책으로, 한 프로세스가 끝날 때까지 다른 프로세스는 계속 기다린다
    - SCHED_RR : Real-Time 정책으로, 프로세스가 서로 번갈아 가며 실행한다
    - SCHED_OTHER : Normal 정책으로,  공평한 시간으로 분배한다 (CFS가 이에 해당한다)

### 응용

ESP32 같이 많이 사용되는 프레임워크에서 사용하는 방식으로 예를 들어 보겠다

```jsx
void main() {
    while(1) {
        Read_Sensor();      
        Calculate_Data();  
        Update_LCD();      
    }
}
```

다음과 같이 함수를 돌며 호출하는 방식은 `Update_LCD` 함수가 일을 처리하는 데에 오랜 시간을 소모할 경우

- `Calculate_Data`
- **`Read_Sensor`**

다음 두 함수는 값을 읽어들일 수 없는 치명적인 문제가 발생함

```jsx
void main() {
    xTaskCreate(Motor_Task, "Motor", 128, NULL, PRIORITY_HIGH, NULL);
    xTaskCreate(LCD_Task, "LCD", 128, NULL, PRIORITY_LOW, NULL);
    vTaskStartScheduler(); 
}
```

다음과 같이 하나의 기능을 Task로 만들고, 이에 따른 우선순위를 부여하여 중요한 일을 먼저 실행할 수 있다
