### Inline 함수에서의 코드 최적화

Inline함수는 함수 호출 시 함수의 주소로 Jump하는 것이 아니라, main에서 계속 flow가 움직인다

즉, 함수 호출 시 함수 자체가 해당 코드 위치로 온다는 개념

- Context Switching
    - 일반 함수는 함수 호출 시 함수의 주소로 Jump하고, 실행이 끝나면 다시 원래 위치로 돌아오는데, 이는 clock을 소모한다
- inline함수의 장단점
    - flow가 움직이지 않기에 클럭은 아낄 수 있지만, 여러 번 호출 시 메모리에 부담이 간다
- 레지스터 vs 라이브러리
    - 라이브러리 함수는 Jump를 하기에 클럭을 낭비함, 이는 레지스터를 직접 제어함으로 해결할 수 있다

### 메모리 구조와 물리적 데이터 통신

오실로스코프로 어떤 신호의 파형을 측정할 때 이에 정렬 순서, 즉 엔디안이 중요하게 작용한다

⇒ 4바이트 정수 (0x12345678) 을 저장할 때

- MSB (0x12) 부터 저장 : 빅 엔디안
    
    → 왼쪽 최상위 비트부터 찍으며 순서대로 나간다
    
- LSB (0x78) 부터 저장 : 리틀 엔디안
    
    → 오른쪽 최하위 비트부터 찍으며 역순서로 나간다
    

| 0 | 1 | 3 | 4 |
| --- | --- | --- | --- |
| 0x12 | 34 | 56 | 78 |
| MSB |  |  | LSB |

**기준**

보드 : 16MHz

Baud : 9600Mz

⇒ 1/9600 = 1초 안에서 한 클럭 

- 레지스터 제어 : 62.5ns가 걸림
- 라이브러리 제어 : 104us가 걸림

### 통신 속도와 하드웨어 클럭

임베디드 분야에서 통신은 매우 중요하게 작용한다

데이터시트의 보드 전용 Clock과 Baud 값의 오차로 데이터 소실을 야기 할 수 있다

- 보드 클럭 : 16MHz
- baud rate : 17MHz

⇒ 다음과 같은 경우에 전송 속도가 보드 클럭을 넘음으로, 당연히 손실이 발생한다

⇒ 송신 측과 수신 측의 클럭 오차가 일정 수준(보통 2~3%)을 넘어가면, 수신 측이 비트를 읽는 타이밍이 엇갈려 데이터를 쓰레기 값으로 인식하거나 소실하게 된다

### 고급 주변장치와 비동기 I/O

- 버퍼(Buffer)와 인터럽트(Interrupt)
    
    ⇒ 일반적인 아두이노(AVR)나 많은 MCU에서 Serial.printf (또는 print)는 DMA를 바로 쓰지 않고 소프트웨어 링 버퍼(Ring Buffer) 를 사용한다.
    
    ⇒ 데이터를 즉시 송신하는 게 아니라 메모리(버퍼)에 복사만 해두고 함수는 바로 종료됩니다. 그래서 실행 시간이 거의 걸리지 않는 것처럼 보인다 (실제 전송은 백그라운드에서 하드웨어 인터럽트가 한 바이트씩 처리한다)
    
    ⇒ 즉 DMA에게 명령만 전달하고 CPU는 다시 본 할일을 하기에 Clock 차이가 많이 나지 않을 수 있다
    
- I/O blocking
    
    ⇒ Blocking 방식과 Non-blocking방식의 차이
    
    - Block
        
        ⇒ 한 함수 분기를 종료까지 기다림
        
    - Non-Block
        
        ⇒ 한 함수 호출 후 DMA, 버퍼에게 전달 후 다시 돌아와 바로 다른 함수 실행
