### Lock-Free

기존 상호배제의 문제점은 Lock을 얻지 못한 스레드는 Lock는 Release 할때까지 대기 상태에 머문다는 단점이 있다. 또 DeadLock과 같은 예상치 못한 문제가 생길 수 있다

Lock 사용 시 공유 메모리에 접근하는 멀티쓰레드가 많아질수록 성능이 오히려 떨어지는 문제가 생긴다

- 기본적으로 Non-blocking이 보장되어야 한다
    
    ⇒ 항상 어떤 스레드로 Wait하지 않고 모든 쓰레드가 실행 중이며, 해당 쓰레드의 CPU 사이클 내에 어떤 작업이 완료될 것이라고 보장한다
    

**Lock-Free는 기본적으로**

⇒ 공유 메모리 변수의 값을 일단 변경하고 해당 값이 맞는지 확인하는 방법이다

⇒ 값을 변경하는 동안 다른 스레드에서 간섭이 있었는지 확인하고, 간섭이 있었다면 그에 대한 처리를 해준다

- CAS (Compare And Swap - 읽고 비교하고 쓰는) 연산으로 Atomic하다
- 락 없이도 최소한 하나의 스레드는 작업의 완료를 보장한다
    - 락을 사용하지 않기에 교착 상태가 없다
    - 자원 점유 방식으로, wait 상태 스레드가 없다

### Wait-Free

모든 스레드가 유한한 시간 내에 작업을 완료할 수 있다는 것을 보장한다

⇒ Lock-Free 보다 더 강력한 동시성을 보장한다
